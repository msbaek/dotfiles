# 오프라인 코드 리뷰 워크스루 설계

현재 브랜치의 변경사항을 분석하여 오프라인 코드 리뷰용 워크스루를 설계합니다.

**base branch**: $ARGUMENTS에서 첫 번째 인자 (기본값: main)
**base commit**: --base-commit 옵션이 있으면 해당 해시부터

---

## 즉시 실행: 1단계 — 변경 범위 식별

다음 명령들을 **지금 바로 실행**하라:

```bash
# 1. base branch 대비 이 브랜치 고유 커밋 식별
git cherry -v <base-branch> HEAD [base-commit] | grep '^+'

# 2. base branch 대비 변경 파일 전체 목록 + 통계
git diff --name-only <base-branch>..HEAD | sort
git diff --stat <base-branch>..HEAD | tail -1

# 3. 고유 커밋 수
git cherry -v <base-branch> HEAD [base-commit] | grep -c '^+'
```

결과를 사용자에게 보여주고 다음을 보고하라:
- 총 고유 커밋 N개
- 총 변경 파일 N개 (+X / -Y 라인)
- 커밋들을 기능별로 대략 분류한 테이블

---

## 즉시 실행: 2단계 — 변경 코드 전체 분석

**서브에이전트 2개를 병렬로 실행**하여 변경된 코드를 전부 읽어라:

### 에이전트 A: 변경 파일 전체 분석
변경된 모든 소스 파일(test 포함)을 읽고 다음을 정리:
- 각 파일의 핵심 역할 (1줄)
- 주요 데이터 흐름
- 테스트가 검증하는 시나리오

### 에이전트 B: 보안/위험 검증
변경된 코드에서 다음을 검증:
- SQL UPDATE/UPSERT에 민감 필드(개인정보, 마스킹 대상)가 포함되었는지
- 암호화/복호화 경로가 올바른지
- 기존 코드와의 호환성 문제가 있는지

> 프로젝트에 마스킹/암호화가 없으면 에이전트 B는 일반적인 코드 품질 검증으로 대체.

---

## 3단계 — 사용자와 대화 (한 번에 하나씩 질문)

분석 결과를 사용자에게 보고한 후, 다음을 **한 번에 하나씩** 질문하라:

**Q1. 리뷰 범위**
> 변경된 N개 파일 전부가 리뷰 대상인가요? 특정 기능만 집중할 부분이 있나요?
> (변경 내용을 기능별로 분류한 테이블을 함께 보여주며 질문)

**Q2. 참석자 수준**
> 리뷰 참석자들이 이 프로젝트에 얼마나 익숙한가요?
> (A) 잘 아는 팀원 (B) 섞여있음 (C) 처음 보는 사람

**Q3. 리뷰 시간**
> 예상 리뷰 시간은?
> (A) 1시간 이내 (B) 1~2시간 (C) 2시간+

**Q4. 발표 방식**
> 어떤 방식으로 워크스루를 진행하시겠어요?
> (A) IDE 디버거로 테스트 실행하며 따라가기
> (B) 코드를 읽으면서 설명
> (C) 테스트 결과(approved.txt 등)를 보여주며 입출력 흐름 설명

**Q5. 비중 배분** (기능 그룹이 2개 이상일 때만)
> 기능별 비중을 어떻게 배분할까요?

---

## 4단계 — 세션 설계 + 접근법 제안

사용자 답변을 종합하여 **2~3가지 워크스루 접근법**을 제안하라.
각 접근법에 장단점 테이블을 포함하고 추천을 밝혀라.

사용자가 접근법을 선택하면, 구체적인 세션 구성을 설계:

각 세션에 포함할 내용:
- **세션명 + 시간** (10~25분)
- **태그 범위**: `walkthrough/<from>` → `walkthrough/<to>`
- **IntelliJ 조작**: 두 태그 선택 → Show Files Diff
- **디버거 breakpoint** (디버거 방식 선택 시): 파일명:라인번호, 확인 사항
- **커버하는 파일 목록**

세션 구성 원칙:
- 가장 중요한/위험한 변경을 먼저
- 유사한 패턴은 하나만 자세히, 나머지는 "동일 구조" 언급
- **전체 커버리지 매트릭스**: 세션별 파일 수 합산 = 총 변경 파일 수 (누락 확인)

사용자에게 세션 구성을 보여주고 승인을 받아라.

---

## 5단계 — 태그 생성

승인된 세션 설계를 기반으로 **태그를 즉시 생성**하라:

```bash
git tag walkthrough/<session-name> <commit-hash>
```

태그 생성 후 **검증을 반드시 실행**:
```bash
# 각 구간 diff 파일 수 확인
git diff --stat walkthrough/<from>..walkthrough/<to> | tail -1

# 전체 합산 = 총 변경 파일 수 확인
```

주의사항:
- 태그는 **현재 브랜치의 커밋**에만 걸어야 IntelliJ에서 보임
- rebase 이전 커밋에 걸면 IntelliJ Git Log에서 안 보임
- 생성 후 사용자에게 IntelliJ에서 태그가 보이는지 확인 요청

---

## 6단계 — 설계 문서 작성

`docs/` 하위에 워크스루 설계 문서를 작성하라.
(경로는 프로젝트 관례에 따름. 없으면 `docs/code-review-walkthrough.md`)

### 문서 필수 포함 내용:
- 개요 (브랜치, 리뷰 형식, 시간, 참석자, 핵심 관심사)
- 변경 범위 요약
- 보안/품질 검증 결과 (해당 시)
- **세션별 구성**: 각 세션마다
  - 태그 범위 + IntelliJ diff 설명
  - breakpoint 또는 핵심 코드 위치
  - 커버하는 파일 목록
- **워크스루 태그 참조표**: 세션 | From 태그 | To 태그 | 변경 파일 수 | 설명
- **전체 커버리지 매트릭스**: 세션별 파일 수 누적 = 총 파일 수
- 준비물 체크리스트
- IntelliJ 조작법

### IntelliJ 조작법 (문서에 포함):
```
1. Git 탭 (Alt+9) → Tags → walkthrough 펼치기
2. From 태그 클릭 → To 태그 Ctrl+클릭 (두 태그 선택)
3. 우클릭 → "Show Files Diff"
→ Changed Files 목록 + Side-by-side diff 표시
```

---

## 7단계 — 커밋

/commit 스킬을 사용하여 설계 문서를 커밋하라.

---

## 핵심 원칙

- **누락 없이**: base branch 대비 변경된 **모든** 파일이 세션에 포함되어야 함
- **태그 기반**: 커밋 해시 대신 의미 있는 태그명으로 navigation
- **즉시 실행**: "이렇게 하세요" 설명이 아니라 직접 명령을 실행하라
- **서브에이전트 활용**: 코드 분석은 서브에이전트로 병렬 실행하여 메인 컨텍스트 보호
- **대화형 설계**: 세션 구성은 사용자와 한 번에 하나씩 질문하며 결정

## 워크스루 종료 후 태그 정리

```bash
git tag -l 'walkthrough/*' | xargs git tag -d
```
